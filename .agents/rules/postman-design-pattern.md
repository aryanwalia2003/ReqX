This project must follow a very strict and consistent code organization philosophy so that development remains extremely fast, easy to reason about, and simple for automated agents to modify. The most important principle is that every file should be small, descriptive, and responsible for only one clear concept. Files must remain short and readable; ideally every file should remain under forty lines of code. If a file grows beyond that, it should immediately be split into smaller files. The goal is that any developer or automated agent can open a file and instantly understand its purpose without scanning large blocks of code. This approach prioritizes speed of navigation, clarity of intent, and modular development.

The project should follow a shallow and clearly defined directory structure. The repository should contain a cmd directory that contains the CLI entrypoint, and an internal directory where the application logic resides. Inside internal, each logical subsystem should have its own folder. These subsystems will include areas such as the runner engine, HTTP execution, websocket support, socket.io client logic, scripting runtime, collection management, and runtime context. Each subsystem folder should only contain files that belong to that specific domain. No cross-domain logic should be mixed into the same file or folder.

Within each subsystem folder, files must be named based on the exact type of code they contain. The naming convention must be explicit and consistent so that an agent can infer the contents of a file purely from its filename. For example, if a file defines a struct, the filename should end with _struct.go. If a file defines an interface, the filename must end with _iface.go. If a file contains methods related to a specific struct or component, the filename must end with _method.go. Constructor functions should live in files ending with _ctor.go. Validation logic should live in files ending with _validate.go. Parsing logic should live in files ending with _parse.go. Helper utilities should live in files ending with _util.go. This rule ensures that file purposes remain predictable and searchable.

Struct definitions should always live in dedicated files. For example, if there is a request model, it should live in request_struct.go. If there is a collection model, it should live in collection_struct.go. These struct definition files should only contain struct definitions and minimal related comments. They should not contain execution logic, helper methods, or unrelated functions. This strict separation ensures that the data model layer remains clean and easy to inspect.

Interfaces should also live in their own files using the _iface.go naming pattern. For example, if there is a request executor abstraction, it should live in executor_iface.go. Interface files should only define interfaces and short comments explaining the contract. They should not contain implementations. The purpose is to keep abstractions isolated and easy to understand.

Implementation logic should live in method files named according to the component they operate on. For example, methods implementing a runner could live in runner_method.go, while HTTP execution logic might live in http_executor_method.go. Each method file should contain only closely related methods. If the number of methods grows too large, they should be split across multiple files grouped by responsibility.

Constructors should always be defined separately from struct definitions. For example, if a runtime context struct exists, its constructor should live in runtime_context_ctor.go. Constructor files should contain small factory-style functions responsible for initializing structs with sensible defaults. This separation helps keep struct files minimal and keeps initialization logic centralized.

The project should strongly follow the Single Responsibility Principle. Each struct, function, or module should have one clearly defined purpose. If any file begins to accumulate multiple responsibilities, it must be split immediately. Short files are not a problem in this architecture; in fact they are encouraged. Many small files are preferred over fewer large files because they improve discoverability and reduce cognitive load when navigating the codebase.

Naming conventions should be extremely explicit. Avoid generic names such as util, manager, handler, or helper unless the file truly represents generic utilities. Prefer descriptive names that reveal intent. For example, instead of naming something manager.go, it should be named request_executor_method.go or collection_loader_method.go depending on its responsibility. This helps automated agents quickly understand the structure of the codebase and locate relevant files.

Functions should remain short and focused. Ideally functions should be under fifteen lines. If a function grows beyond that, it should be broken down into smaller functions with descriptive names. Long procedural blocks should be avoided because they make reasoning about code harder. Small composable functions make both human and automated modifications much safer.

The runner subsystem should remain isolated and treated as the orchestration engine of the system. It should only coordinate execution flow and should not contain transport-specific logic. Transport logic should live in separate modules such as HTTP executors, websocket clients, and socket.io clients. The runner should only call these modules through clearly defined interfaces.

Similarly, the scripting subsystem should be isolated from request execution logic. Script execution should interact with the runtime context but should not directly manipulate transport layers. This separation keeps the architecture clean and prevents tight coupling between components.

The runtime context should act as the shared state container for variables, environment values, and script outputs. It should be designed as a simple struct with clearly defined methods for reading and writing variables. The context must remain lightweight and should not embed heavy dependencies.

Imports across subsystems should be minimal and directional. Lower-level modules such as transport executors should never depend on the runner layer. The dependency flow should always move upward toward orchestration. This prevents circular dependencies and keeps the system easier to maintain.

Logging and error handling should be straightforward and consistent. Avoid deeply nested error flows. Functions should return errors early and clearly. Error messages should be descriptive and contextual so that CLI output remains informative during request execution.

Finally, the most important guiding principle for this project is development velocity. The architecture must remain simple enough that features can be implemented quickly without extensive refactoring. If a design choice slows down development or introduces unnecessary complexity, it should be reconsidered. The project is intended to be a fast-moving internal tool, so clarity, simplicity, and modularity should always take precedence over theoretical architectural perfection.
